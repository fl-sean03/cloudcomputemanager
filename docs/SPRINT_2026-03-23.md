| `agents/` | 1,086 | ✅ Complete | Async SDK for AI agents |

**Hardened 2026-03-21**: Critical bugs fixed (recovery commit, daemon commits, stdio redirect, SSH retry, security hardening).

### What Failed in Production (MXene Campaign)

CCM was **abandoned mid-campaign** in favor of raw `vastai` CLI + SSH scripts. The workflow needed:

```
Find instance → Create → Install LAMMPS → Upload files → Run NPT (8hrs) →
Poll progress → Extract dimensions → Run NVT deform → Run 22 shear jobs → Download → Terminate
```

**CCM could only handle**: steps 1-2 (find/create) and final download/terminate. Everything in between required direct SSH because:

1. **No setup commands** — Can't `apt install lammps` before the job runs
2. **Hard-coded 300s provisioning timeout** — Large images fail
3. **No mid-flight interaction** — Can't check progress, upload files, or chain commands
4. **No progress visibility** — 8-hour jobs are black boxes
5. **No multi-stage pipelines** — NPT→NVT→shear can't be one job
6. **No variable substitution** — Duplicate YAMLs for each structure
7. **No batch parameter sweeps** — 22 shear runs need manual YAML generation

### What Good Looks Like (Research Findings)

| Pattern | Best-in-class | CCM Gap |
|---------|--------------|---------|
| Setup commands | SkyPilot `setup:` field in YAML | No setup phase |
| Multi-stage | SkyPilot `---` YAML doc separators | Single command only |
| Variables | SkyPilot `${VAR}` + `--env` CLI | No substitution |
| Progress | CCM already has `JobMetrics` model | Not wired to LAMMPS log parsing |
| SSH access | All tools expose direct SSH | CCM abstracts it away |
| Cost tracking | Custom per-project databases | No historical tracking |
| Benchmarking | Manual (see MXene benchmarks/) | No automation |

## Design Decisions

### Decision 1: Direct SSH vs SSH Passthrough

**Decision: Give agents direct SSH credentials instead of passthrough.**

Rationale: CCM's `execute_command()` adds a layer that provides retry logic but prevents interactive use, streaming, and mid-flight file transfers. The agent (or user) benefits more from having the raw SSH connection info (host, port, key) and using SSH directly. CCM should:
- Store and expose SSH connection details per instance
- Provide convenience wrappers (`ccm exec`, `ccm ssh`) for humans
- Let the Agent SDK return SSH credentials so agents can use `asyncio.create_subprocess_exec("ssh", ...)` directly
- Keep the retry logic in the convenience wrappers, not as a mandatory layer

### Decision 2: Benchmark Suite Architecture

**Decision: CCM-native benchmark framework with per-project benchmark configs.**

Based on the MXene benchmark methodology (`benchmarks/run_comparison.py`):
- User defines a `benchmark.yaml` with: workload command, input files, GPU types to test, metrics to extract via regex
- Fully workload-agnostic — works for LAMMPS, NAMD, GROMACS, PyTorch, JAX, or any CLI tool
- `ccm benchmark run benchmark.yaml` provisions instances, runs short benchmarks, collects results
- `ccm benchmark results` shows cost-performance matrix
- Results persist in SQLite for historical comparison
- Pre-flight validation (`core/validation.py`) already extracts performance metrics — extend this

### Decision 3: What to Deprioritize

Per the MXene feedback doc and audit:
- **PackStore**: Over-engineered. `apt install` + setup scripts cover real needs. Keep registry but don't expand.
- **REST API**: No current consumer. Keep it working but don't add features.
- **S3 sync**: Not needed at our scale. rsync to local is sufficient.
- **Multi-cloud**: Vast.ai only for now. Keep the ABC but don't add providers.
- **CRIU checkpointing**: Not practical for most scientific workloads. Application-level checkpoints are correct.

## Sprint Backlog

### Phase 1: Core Workflow Fixes (Unblock Basic Usage)

These 3 items would have been enough to use CCM for the entire MXene campaign.

#### 1.1 Custom Setup Commands
**Priority**: P0 — Critical
**Effort**: Small (Vast.ai already has `--onstart-cmd`)
**Files**: `providers/vast.py`, `core/models.py`, `core/templates.py`

Add `setup` field to job YAML (workload-agnostic — works for any software):
```yaml
image: ubuntu:22.04
setup: |
  apt-get update -qq && apt-get install -y -qq lammps   # or: pip install torch, conda install gromacs, etc.
command: mpirun -np 16 lmp -in simulation.inp
```

Implementation:
- Add `setup_commands: Optional[str]` to Job model
- Pass `setup` value as `--onstart-cmd` to `vastai create instance`
- In `wait_for_ready()`, after SSH is available, verify setup completed (check for sentinel like `SETUP_DONE` or poll for the installed binary)
- Templates should support `setup` field

#### 1.2 Configurable Provisioning Timeout
**Priority**: P0 — Critical
**Effort**: Tiny
**Files**: `providers/vast.py`, `core/models.py`

```yaml
provisioning:
  timeout: 600          # seconds (default: 300)
  retry_on_timeout: true
  max_retries: 2
```

Implementation:
- Add `provisioning_timeout: int = 300` to Job config
- Pass to `wait_for_ready()` instead of hardcoded 300
- Add retry loop if `retry_on_timeout` is set

#### 1.3 SSH Passthrough / Direct Access
**Priority**: P0 — Critical
**Effort**: Medium
**Files**: `cli/main.py`, `cli/instances.py`, `providers/vast.py`, `agents/sdk.py`

Expose SSH credentials and add convenience commands:
```bash
ccm exec <job_id> "tail -5 output.log"          # One-shot command
ccm upload <job_id> ./local/file /remote/path    # Upload mid-flight
ccm ssh <job_id>                                  # Interactive shell (existing, verify it works by job_id)
```

For Agent SDK:
```python
creds = await agent.get_ssh_credentials(job_id)
# Returns: SSHCredentials(host, port, key_path, username)
# Agent uses these directly for subprocess SSH calls
```

Implementation:
- Add `get_ssh_credentials()` to Agent SDK that returns host/port/key
- Add `ccm exec <job_id>` CLI command (looks up instance, runs SSH command)
- Add `ccm upload <job_id> <local> <remote>` CLI command (rsync single file)
- Ensure `ccm ssh <job_id>` works (currently only by instance_id)

### Phase 2: Visibility & Monitoring

#### 2.1 Pluggable Progress Parsing
**Priority**: P1 — High
**Effort**: Small (detector infrastructure exists)
**Files**: `daemon/monitor.py`, `core/models.py`

Workload-agnostic progress monitoring via configurable parsers:
```bash
ccm jobs status <id>
# Job: my-job  Status: RUNNING  Progress: 3.8M/5M (76%)  Rate: 185 units/s  ETA: 1.2h
```

Implementation:
- Add configurable progress parsing to job config:
  ```yaml
  progress:
    type: regex_parse        # or file_growth, custom_command
    file: /workspace/output.log
    regex: '^\s*(\d+)\s+'   # Extract first number (generic — works for any line-based output)
    total: 5000000
  ```
- Supported types:
  - `regex_parse`: SSH tail the file, apply regex, compute current/total
  - `file_growth`: Existing behavior — track output file size growth
  - `custom_command`: Run arbitrary command on instance, parse stdout as progress number
- In `JobMonitor.check_job_health()`, call progress parser and update `JobMetrics`
- Display in `ccm jobs status` and `ccm jobs list`
- Built-in regex presets for common workloads (LAMMPS, NAMD, PyTorch epoch) but user can specify any regex

#### 2.2 Notification Hooks
**Priority**: P1 — High
**Effort**: Small
**Files**: `daemon/monitor.py`, `core/config.py`

Add webhook/command notifications for key events:
```yaml
# In CCM config or job YAML
notifications:
  on_complete: "curl -X POST https://hooks.slack.com/... -d '{\"text\": \"Job ${JOB_NAME} completed\"}'"
  on_failure: "curl -X POST https://hooks.slack.com/..."
  on_budget_exceeded: "echo 'Budget exceeded for ${JOB_NAME}' >> ~/ccm_alerts.log"
```

Implementation:
- Add `notifications` dict to job config
- In daemon event handlers, shell out to notification command with variable substitution
- Support both webhook URLs and arbitrary shell commands

### Phase 3: Multi-Stage & Batch Workflows

#### 3.1 Variable Substitution in Job YAML
**Priority**: P1 — High
**Effort**: Small
**Files**: `core/templates.py`, `cli/jobs.py`

```bash
ccm jobs submit job.yaml --set STRUCTURE=F100_dry --set DATA_FILE=mxene_dry_F100.data
```

Implementation:
- In `load_config_with_template()`, after loading YAML, run `string.Template` substitution
- CLI `submit` gets `--set KEY=VALUE` repeatable option
- Variables available: user-provided + built-in (`$JOB_NAME`, `$TIMESTAMP`, `$RANDOM`)

#### 3.2 Multi-Stage Jobs
**Priority**: P1 — High
**Effort**: Medium
**Files**: `core/models.py`, `daemon/monitor.py`, `cli/jobs.py`

```yaml
stages:
  - name: npt_equilibration
    command: mpirun -np 16 lmp -in npt.inp > npt.log 2>&1
    timeout_hours: 24
    completion_check: grep -q "EQUILIBRATION COMPLETE" npt.log

  - name: extract_dimensions
    command: python3 extract_mean_dims.py npt.log
    timeout_hours: 0.1

  - name: shear_sweep
    command: bash run_all_shear.sh
    timeout_hours: 12
```

Implementation:
- Add `stages: list[JobStage]` to Job model (JobStage: name, command, timeout, completion_check)
- Add `current_stage: int = 0` to Job model
- Daemon monitor: when `.ccm_exit_code` detected, check if more stages remain
  - If yes: advance `current_stage`, run next stage command via SSH, reset exit code sentinel
  - If no: mark job COMPLETED
- `ccm jobs status` shows current stage name and progress
- Each stage can have its own `progress` config

#### 3.3 Batch Parameter Matrix
**Priority**: P2 — Medium
**Effort**: Small
**Files**: `cli/batch.py`, `core/templates.py`

```yaml
# shear_sweep.yaml
template: shear-job.yaml
matrix:
  STRESS: [50, 75, 100, 125, 150, 175, 200, 225, 250, 275, 300]
  DIRECTION: [X, Y]
```

```bash
ccm batch submit shear_sweep.yaml  # Creates 22 jobs (11 × 2)
```

Implementation:
- In `batch_submit()`, detect `matrix` key
- Compute Cartesian product with `itertools.product`
- For each combination, substitute variables and submit as individual job
- Track all as a batch with shared project name

### Phase 4: Benchmark Framework

#### 4.1 Benchmark Suite Engine
**Priority**: P2 — Medium
**Effort**: Medium
**Files**: NEW `benchmarks/` module

```yaml
# benchmark.yaml — fully workload-agnostic
name: my-workload-bench
description: Cost-performance comparison across GPU tiers

workload:
  image: ubuntu:22.04
  setup: |
    apt-get update -qq && apt-get install -y -qq lammps openmpi-bin  # or: pip install torch, etc.
  command: mpirun --allow-run-as-root -np ${NCPUS} lmp -in ${INPUT_FILE}
  # command: python train.py --epochs 1 --batch-size ${BATCH_SIZE}  # ML example
  files:
    - benchmarks/benchmark.data
    - benchmarks/lammps_benchmark.inp

matrix:
  gpu_type: [RTX_3060, RTX_4070, RTX_4090]

metrics:  # User defines what to extract — any regex on stdout/stderr/file
  - name: primary_metric
    source: stdout
    regex: 'Performance:.*?([\d.]+)\s+timesteps/s'  # or: 'throughput:\s+([\d.]+)\s+samples/sec'
    unit: ts/s   # Purely descriptive label

repetitions: 3  # runs per configuration
timeout: 300    # seconds per run

budget:
  max_per_instance: 0.50
```

```bash
ccm benchmark run benchmark.yaml              # Run full suite
ccm benchmark run benchmark.yaml --gpu RTX_3060  # Single GPU type
ccm benchmark results                         # Show all results
ccm benchmark results --sort cost_per_mts     # Sort by cost per million timesteps
ccm benchmark compare benchmark1 benchmark2   # Compare two runs
```

Implementation:
- New `src/cloudcomputemanager/benchmarks/` module:
  - `engine.py` — Orchestrate benchmark runs (provision → setup → run → collect → destroy)
  - `models.py` — BenchmarkSuite, BenchmarkRun, BenchmarkResult SQLModel tables
  - `analysis.py` — Cost-performance analysis, ranking, comparison
  - `reports.py` — Rich table output, CSV export
- CLI commands in `cli/benchmarks.py`
- Persist results to same SQLite database
- Reuse existing `VastProvider` for instance lifecycle
- Reuse `core/validation.py` metric extraction patterns

#### 4.2 Historical Cost Tracking
**Priority**: P2 — Medium
**Effort**: Small
**Files**: `core/models.py`, `daemon/monitor.py`

Every completed job automatically records:
```
(gpu_type, cpu_cores, workload_type, system_size, steps_per_second, hourly_rate, total_cost, cost_per_million_steps)
```

- New `CostRecord` SQLModel table
- Daemon writes a record on job completion
- `ccm costs show` displays historical data
- `ccm costs recommend --project my-project` suggests best GPU based on historical data

### Phase 5: Instance Management

#### 5.1 Instance Reuse Across Jobs
**Priority**: P2 — Medium
**Effort**: Medium
**Files**: `core/models.py`, `providers/vast.py`, `cli/instances.py`, `cli/jobs.py`

```bash
ccm instances reserve <instance_id> --hours 8    # Keep alive after job ends
ccm jobs submit job.yaml --instance <instance_id>  # Run on existing instance
```

Implementation:
- Add `RESERVED` state to `InstanceStatus`
- Add `reserved_until: Optional[datetime]` to Instance model
- `submit_job()` with `--instance` skips provisioning, uses existing instance
- Daemon checks `reserved_until` and terminates expired reservations

#### 5.2 Instance Pool
**Priority**: P3 — Low
**Effort**: Large
**Files**: Multiple

Auto-reuse: when a job completes on a reserved instance and another job in the same project is queued, assign it to the same instance. This is an optimization for multi-stage workflows where explicit `--instance` is too manual.

## Test Case: MXene Campaign Replay

With all sprint features implemented, the MXene Campaign 2026 workflow becomes:

```bash
# 1. Benchmark to find best GPU (one-time)
ccm benchmark run benchmarks/mxene-lammps.yaml
# → RTX 2080 Ti: 100 ts/s @ $0.08/hr = $0.80/M-ts (BEST VALUE)
# → RTX 4090:   200 ts/s @ $0.25/hr = $1.25/M-ts

# 2. Submit relaxation (one command per structure)
ccm jobs submit mxene_relaxation.yaml \
  --set STRUCTURE=F100_dry \
  --set DATA_FILE=mxene_dry_F100.data \
  --set COEFFS=dimfix_v3_F100.inc

# 3. Monitor
ccm jobs list --project mxene-campaign
# ID   NAME              STAGE           PROGRESS      RATE     ETA
# 42   relax-F100_dry    npt (1/3)       3.8M/5M 76%  185ts/s  1.2h
# 43   relax-OH-Only     npt (1/3)       0.9M/5M 17%  180ts/s  6.1h
# 44   relax-OH50-F50    npt (1/3)       1.5M/5M 31%  190ts/s  5.1h

# 4. Submit shear sweep after relaxation completes
ccm batch submit shear_sweep.yaml
# → 22 jobs created (11 stresses × 2 directions)

# 5. Check on it later
ccm batch status --project mxene-campaign
# Shear sweep: 18/22 complete, 4 running, 0 failed
# Total cost: $3.47
```

Three commands for the full relaxation. One command for the shear sweep. Full visibility throughout.
